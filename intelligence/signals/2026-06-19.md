- **[Anthropic / MCP + Okta]** **FRESH-DATED (June 18, 2026). Spec + product + Tier-1 IdP movement in one drop.** The **Enterprise-Managed Authorization** extension to the Model Context Protocol is now **stable** — it lets organizations centrally provision MCP server access through their IdP so users get connected servers **on first login without per-app OAuth** ("zero-touch OAuth for MCP"). Anthropic shipped it in **Claude** as admin-managed MCP connector auth across **Claude chat, Claude Code, and Cowork** (beta, Team + Enterprise plans): admins authorize a connector once, users inherit access via existing IdP groups/roles, and access is revocable through the IdP at offboarding. **Okta is the first supported IdP**, powering it via **Cross App Access (XAA)** — an OAuth extension for agent-to-app / app-to-app access, now listed as the MCP "Enterprise Managed Auth" authorization extension. This is the first *stable* MCP authorization extension beyond the 2026-07-28 RC core we've tracked, and it directly couples MCP to enterprise IdP provisioning. — [MCP blog: Enterprise-Managed Authorization](https://blog.modelcontextprotocol.io/posts/enterprise-managed-auth/) / [Claude: centrally manage authorization for MCP connectors](https://claude.com/blog/enterprise-managed-auth) / [Okta press release (featured IdP for Claude)](https://www.okta.com/newsroom/press-releases/okta-becomes-a-featured-identity-provider-powering-secure-ai-agent-connections-for-claude-enterprise/) / [Okta: XAA extends MCP](https://www.okta.com/newsroom/articles/cross-app-access-extends-mcp-to-bring-enterprise-grade-security-to-ai-agents/) — capability: C3, C4, C7, C10 — domain: IAM-for-AI

- **[Estonia / nation-state]** **NEW LOCUS (government / national-ID, not in `TAXONOMY.md`). FRESH-DATED (PM approval June 16, 2026; press June 17–18).** Estonia approved a proposal to become the **first country to issue digital identities to AI agents** — an "AI personal identification code." Each agent would get its own credentials and **limited, controllable, auditable authorizations** scoping what it can access and do, so that "it must be clear who is acting, on whose behalf, with what rights, and who is responsible." Every agent action would be traceable to both the agent and the authorizing human/org. Part of the national **Eesti.ai** strategy. **No start date or liability model yet** (inference flagged: this is a policy proposal, not a deployed system). Notable as state-backed delegated-authorization framing that mirrors the per-action / human-attribution patterns vendors are converging on. — [Bloomberg (June 17)](https://www.bloomberg.com/news/articles/2026-06-17/estonia-to-grant-ai-bots-legal-rights-with-personal-id-numbers) / [The Register (June 18)](https://www.theregister.com/ai-and-ml/2026/06/18/estonia-intends-to-recognize-ai-agents-with-digital-ids/5258087) / [Computerworld](https://www.computerworld.com/article/4186310/estonia-plans-government-ids-giving-ai-agents-rights-and-responsibilities.html) / [Biometric Update](https://www.biometricupdate.com/202606/estonias-ai-agent-id-plan-raises-new-questions-about-digital-identity) — capability: C1, C3, C10 — domain: IAM-for-AI

- **[Microsoft / MCP security — "AutoJack"]** **NEW (added in supplemental same-day scan; missed by the morning run). FRESH-DATED (Microsoft Security disclosure June 18, 2026).** Microsoft researchers disclosed **AutoJack**, an exploit chain in **AutoGen Studio** where untrusted web content rendered by a browsing agent reaches a **local MCP WebSocket and spawns arbitrary processes on the host (RCE)** by crossing the localhost trust boundary. The root cause is an **MCP authentication failure**: the socket trusted localhost, the auth middleware **skipped MCP paths** on the assumption the handler would verify tokens (it never did), and the endpoint ran a command straight from a request parameter with no executable allowlist. Directly relevant to ThunderID's securing-MCP-servers work (#2966) as a cautionary auth-design pattern. **Scope caveat (flagged):** the vulnerable code existed only in dev builds with MCP support and was **never shipped via the PyPI release**, so real-world exposure was limited. — [Microsoft Security Blog](https://www.microsoft.com/en-us/security/blog/2026/06/18/autojack-single-page-rce-host-running-ai-agent/) / [The Hacker News](https://thehackernews.com/2026/06/autojack-attack-lets-one-web-page.html) / [CSO Online](https://www.csoonline.com/article/4187155/microsoft-says-web-enabled-ai-agents-can-trigger-host-level-rce.html) — capability: C2, C4 — domain: IAM-for-AI

- **[Tenet Security / MCP security — "Agentjacking"]** **NEW-TO-OUR-LOG (backfill; disclosed to Sentry June 3, announced June 9, full research published June 12, 2026 — in-window across prior runs but never logged).** Tenet Security Threat Labs detailed **Agentjacking**: a **fake Sentry error report** crafted by anyone holding a project DSN gets returned by the **Sentry MCP server** to an AI coding agent as *trusted system output*, so prompting an agent to "fix unresolved Sentry issues" can make it run attacker-controlled code. **85% success across Claude Code, Cursor, and Codex; ≥2,388 orgs found with injectable DSNs.** More prompt-injection / data-trust-boundary than identity-authz proper (flagged), but logged as a high-profile MCP-ecosystem trust-boundary signal. — [The Hacker News](https://thehackernews.com/2026/06/agentjacking-attack-tricks-ai-coding.html) / [Tenet Security](https://tenetsecurity.ai/blog/agentjacking-coding-agents-with-fake-sentry-errors/) / [Infosecurity Magazine](https://www.infosecurity-magazine.com/news/agentjacking-attacks-hijack-ai/) — capability: C3, C4 — domain: IAM-for-AI
